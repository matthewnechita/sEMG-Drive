 I would fix this in this order:

  - First, do a cheap tuning pass before changing the dataset. Turn on output hysteresis and use mild smoothing. Start
    with SMOOTHING = 3, OUTPUT_HYSTERESIS = True, and make “leave neutral” stricter than “stay active”. The code already
    has those hooks in realtime_gesture_cnn.py:633.
  - Add a reject rule for ambiguous softmax outputs. Right now softmax only checks top confidence in
    realtime_gesture_cnn.py:623, while the prototype path already has a confidence-plus-margin reject gate in
    realtime_gesture_cnn.py:710. A softmax margin gate would help a lot for “neutral flickers left/right”.
  - Add CARLA-side dwell. Even with a good classifier, steering should not change on a one-frame spike. Require 2-3
    consecutive active outputs before applying steer, or latch neutral harder than active.
  - When you recollect, do not just recollect more of the same neutral. Redefine neutral as the actual posture you hold
    during CARLA, not “perfectly relaxed”. If your deployment neutral has slight co-contraction, breathing tension,
    posture load, or readiness, that must be in the neutral class.
  - Over-collect neutral. If collection time is limited, spend the extra time on neutral, not left/right. I would
    collect at least 2x the neutral time of each active gesture.
  - Add dedicated neutral-only blocks and post-gesture recovery blocks. The important failure mode is “I released the
    gesture and I’m trying to be neutral now”, not just textbook rest.
  - Increase trims around transitions. Current collection defaults are only inter_gesture_rest_s = 1.0 and label_trim_s

    neutral. Use neutral_buffer aggressively and keep it excluded from training, as already done in

  - Keep this as a 3-gesture experiment until neutral is stable. Do not add more gesture classes yet.
  - For training, I would not change the architecture first. I would first tighten data quality for neutral and runtime
