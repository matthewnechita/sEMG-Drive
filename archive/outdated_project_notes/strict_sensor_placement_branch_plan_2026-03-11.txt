Strict Sensor Placement Branch Plan
Date: March 11, 2026

Purpose
- Preserve the current salvage-path work in one branch.
- Tomorrow, create a separate branch for a clean fixed-position workflow.
- Compare the two approaches directly instead of mixing them.

Current branch: salvage path
- Keep the current changes in place on this branch.
- Training path:
  - `USE_SENSOR_TYPE_CANONICALIZATION = True`
  - `USE_SINGLE_BLOCK_PERMUTATION_AUGMENTATION = True`
- Realtime path:
  - bundle-driven type canonicalization is enabled for new salvage-trained models
  - output hysteresis was added to reduce realtime flicker
- What this branch achieved:
  - gross channel mismatch / random sensor-number issues were reduced
  - single-arm right and left testing became workable again
- Remaining issue:
  - left-turn vs right-turn directional separation is still not stable enough
  - likely cause: the salvage model is too invariant within the ambiguous single-channel sensor block

New branch to create tomorrow: strict fixed-position workflow
- Goal:
  - remove ambiguity inside the repeated same-type sensors by enforcing a fixed physical layout
  - retrain on clean data with stable channel semantics

Strict fixed-position rules
- Define a fixed position map for each arm.
- Every physical position should have a fixed logical slot.
- Prefer fixed pair numbers / sticker-numbered sensors for each slot.
- If a sensor dies, replacement must inherit the same logical slot and pairing rule.
- Keep sensor type AND within-type position fixed across:
  - data collection
  - realtime testing
  - future recollection sessions

Recommended fixed-position slots
- Right arm:
  - R_Galileo
  - R_Maize
  - R_Mini
  - R_Avanti_1
  - R_Avanti_2
  - R_Avanti_3
- Left arm:
  - L_Galileo
  - L_Maize
  - L_Avanti_1
  - L_Avanti_2
  - L_Avanti_3

Tomorrow branch plan
1. Create a new branch for strict fixed placement.
2. Disable salvage-specific training assumptions in `train_per_subject.py`:
   - `USE_SENSOR_TYPE_CANONICALIZATION = False`
   - `USE_SINGLE_BLOCK_PERMUTATION_AUGMENTATION = False`
3. Keep data collection metadata saving in place.
   - `DataCollector/CollectDataWindow.py` now saves `emg_channel_labels`
4. Realtime:
   - use explicit right/left pair-number membership for dual mode only if the current session mapping is known
   - use explicit within-arm pair order only if the physical order is actually fixed and documented
5. Collect a small clean pilot dataset first.
   - right arm only
   - left arm only
   - fixed physical positions
   - same subject
   - same 3 gestures: `neutral`, `left_turn`, `right_turn`
6. Retrain per-subject right and left models from that clean pilot dataset.
7. Test single-arm realtime before expanding collection volume.

What NOT to do on the strict-position branch
- Do not train on historically mixed-layout data and expect fixed-order models to be reliable.
- Do not randomize same-type sensors among different Avanti positions.
- Do not rely on pair-order lists unless the order is actually tracked and fixed.

Comparison plan
- Branch A: salvage path
  - old data preserved as much as possible
  - type canonicalization + permutation augmentation
- Branch B: strict fixed-position path
  - clean recollection
  - no canonicalization
  - no permutation augmentation
- Compare:
  - offline confusion between `left_turn` and `right_turn`
  - single-arm realtime flicker
  - required contraction intensity
  - day-to-day repeatability

Most likely outcome
- The strict fixed-position branch is the highest-probability path for fixing directional flicker.
- The salvage branch remains useful as a fallback and as a way to preserve value from old data.
